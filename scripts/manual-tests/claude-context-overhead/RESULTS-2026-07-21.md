# Claude first-turn context bill of materials

## Scope

This is the follow-up experiment for NIM-1988. It used the committed harness from `225230bbea4d382e1c10b48bb1ec22e8d85e4ffd`, extended with privacy-safe segment fingerprints, compressed-response usage capture, isolated component profiles, and analysis tooling.

Matched controls for every primary case were Claude Code 2.1.210, `@anthropic-ai/claude-agent-sdk` 0.3.204, resolved model `claude-haiku-4-5-20251001` through the repository's `haiku` alias, `/Users/ghinkle/sources/stravu-editor`, prompt `Reply with only OK.`, ToolSearch on, `dontAsk`, no session persistence, strict empty user MCP configuration, and a $0.25 maximum budget. Every case was dry-run before paid calls. The heavily dirty shared worktree was preserved.

The historical checkpoint remains 43,178 raw versus 56,250 full on Fable, a 13,072-token increment. The current Haiku matrix is not a replacement for that model-specific number. It supplies a byte-level bill of materials for the same profile shape and actual Haiku `contextTokens` for each isolated component. Reconstructing exact Fable token attribution after the fact is not possible from the historical summaries alone.

## Exact `tools_changed` findings

### Dynamic tag counts

Using this real session id made the live workspace tag list available. Removing only the session's existing `ai` tag changed exactly one tool and one leaf:

```text
tool: mcp__nimbalyst__update_session_meta
path: $.input_schema.properties.add.description
tool bytes: 1,726 -> 1,726
leaf bytes: 424 -> 424
tool fingerprint: 79ab75ed09ca82d8bb8a -> b73f6d18f33fd0d6febd
leaf fingerprint: 21a8af506712e28bb7e5 -> 3b16f6148af110a6fc8a
```

Restoring `ai` restored both original fingerprints exactly. Tool order, server order, system blocks, and message segments did not change. This is direct byte-level proof that count-only drift changes the eager tool prefix even when serialized length stays constant.

### MCP readiness and membership

One all-MCP run reached the API while all 21 MCP servers were still `pending`; three later runs had all 21 `connected`.

| State         | CLI init tools | API tools | Request bytes | contextTokens | Deferred-list block 0 |
| ------------- | -------------: | --------: | ------------: | ------------: | --------------------: |
| all pending   |             30 |        11 |       147,011 |        38,722 |           1,435 bytes |
| all connected |            303 |        18 |       170,698 |        46,066 |          14,005 bytes |

The transition added the seven eager core schemas, changed tool and server membership/order fingerprints, and added 12,570 bytes to the deferred listing. System fingerprints stayed identical. In a resumed session, this exact pending-to-connected transition is sufficient to produce a `tools_changed` miss before considering tag counts. It also shows how plugin loading can appear to reverse a comparison: the plugin-only payload cost was just 69 tokens, while the full profile reached the API connected and the earlier all-MCP case was still pending. Plugin startup delay is a plausible contributor, not proven causation, and the timing effect must not be mistaken for plugin savings.

### Ordering and other dynamic fields

Across connected repeats, tool order, server order, every tool schema fingerprint, every system block, and every initial-message segment were byte-stable. ToolSearch-off full runs were also byte-identical with all 302 request tools in the same order.

The request `metadata` option fingerprint changed on every fresh CLI invocation while its byte size stayed constant. This experiment used fresh no-persistence sessions, so that observation is not evidence that metadata invalidates Anthropic's resumed-session cache. It is now visible for a real two-turn proxy test.

## Out-of-band title and tags cost

The proxy observed one invariant auxiliary no-tools request per direct CLI run: 2,300 request bytes, 521 actual context tokens, and 12 output tokens. Its shape is consistent with Claude Code's small built-in title-generation path, but the request is labeled auxiliary because the body is intentionally not logged and the harness cannot prove its semantic purpose.

An explicitly spawned Nimbalyst `claude-code:haiku` child completed read-only with no edits. Title, tags, phase, and `has_been_named` all succeeded. Its two visible main calls were 48,828 and 49,252 context tokens. The SDK result aggregate reported 18 uncached input, 49,244 cache creation, and 48,818 cache read tokens, exactly the sum of those two visible calls; it did not report an additional prefix-sized input/cache request for title or tags. The result reported 350 output tokens while raw assistant usage rows exposed only a two-token floor, so side-control output cannot be decomposed reliably from persisted rows.

Conclusion: a stable 521-token auxiliary request is measured, but a separate tens-of-thousands-token out-of-band naming request was not reproduced in this controlled child. Proving or refuting that larger cost requires routing the live Nimbalyst SDK transport through the proxy or instrumenting SDK control requests directly.

## Live Agent SDK cache-seam validation

The follow-up gate routed real spawned `claude-code:haiku` Nimbalyst sessions through the same proxy by temporarily setting `ANTHROPIC_BASE_URL` in the Claude Code user environment. The prior empty environment map was restored immediately after every run. Four read-only children completed with no edited files. The final harness groups a shared proxy by a process-scoped HMAC of model plus request metadata, so concurrent Sonnet traffic no longer contaminates the child's comparisons.

### System cache-control TTL oscillation

The same first-turn pattern reproduced across all four children. The final repeat added per-leaf system fingerprints and isolated the only changes:

| Main request        | System cache-control state |  input | cache creation | cache read | contextTokens |
| ------------------- | -------------------------- | -----: | -------------: | ---------: | ------------: |
| initial             | blocks 1/2 `ttl: 1h`       |     10 |         48,687 |          0 |        48,697 |
| after Read          | blocks 1/2 default 5m      | 22,477 |         26,538 |          0 |        49,015 |
| after metadata tool | blocks 1/2 `ttl: 1h`       |      8 |            571 |     48,687 |        49,266 |

For the middle request, system blocks 1 and 2 each shrank by exactly 11 bytes. The only changed leaf in both blocks was `$.cache_control.ttl`, removed from `1h` to the API's default 5-minute ephemeral policy. Tool schemas, tool order, server order, request options, and all other system leaves were stable. Restoring `ttl: 1h` restored the 48,687-token cache read.

Two invariant auxiliary Haiku requests occurred during the first turn, one before the initial main request and one between the initial and middle main requests. Each cost 626 context tokens in the grouped repeat. Their interleaving is consistent with the live title/control path influencing cache policy, but this experiment does not prove causation. The child requests did not include Anthropic's optional diagnostics field, so they exposed actual cache reads/creation rather than a `cache_miss_reason` label.

Two earlier live children showed the same full-rewrite/warm-next-turn sequence even when the schema length stayed constant and only the count bytes changed.

Request metadata was stable throughout each resumed child, resolving the earlier uncertainty for this path: it did not cause these within-session misses.

### Static metadata-tool schema implementation

The first cache-stability fix is now implemented in source. `update_session_meta` no longer queries workspace sessions or embeds mutable tag names, counts, or count-derived ordering in its eager schema. Its tag guidance is fixed text, and the system addendum now says to reuse existing workspace tags "when known."

The source-built schema object is 1,452 bytes with a 165-byte raw `add.description` string. After restart, the wire tool was 1,469 bytes and its JSON-string leaf was 167 bytes, versus 1,727 and 425 bytes in the final pre-fix live run: a like-for-like wire reduction of 258 bytes at both the tool and changed-leaf levels. This is a structural measurement, not an actual-token savings claim.

A pre-restart validation attempt correctly detected the old 1,727-byte schema and was excluded. The post-restart child served the new 1,469-byte schema with the expected fixed leaf fingerprint.

### Redundant tags/phase side-request implementation

The next isolated change removes Claude Agent's out-of-band `askSideQuestion` request for tags and phase. The required `update_session_meta` call remains the authoritative tags/phase path, the system prompt still says tags and phase are not assigned out-of-band, and built-in `generateSessionTitle` remains enabled for the session name. The default planning-phase fallback is retained when the metadata tool has not set a phase.

Three post-restart Haiku children completed read-only with no edited files. The exact sequential repeat produced the original request order—initial request, `Read` result, metadata result—and kept both cached system blocks at `ttl: 1h` throughout:

| Main request        | System changes | cache creation | cache read | contextTokens |
| ------------------- | -------------- | -------------: | ---------: | ------------: |
| initial             | none           |         22,189 |     26,458 |        48,657 |
| after `Read`        | none           |          6,884 |     48,647 |        55,539 |
| after metadata tool | none           |            426 |     55,531 |        55,967 |

The initial request reused a cross-session 26,458-token prefix, so it was not a cold-start size measurement. The within-session result is decisive: system, tool, tool-order, server-order, and relevant option fingerprints remained unchanged, and each continuation read the complete prior prefix. The 6,884-token creation after `Read` is the newly appended file/tool history, not a prefix rewrite.

The resumed tag-mutation child also kept every tool and system fingerprint stable. Before the mutation it read 49,033 tokens and created 92; immediately after changing `ai` to `research` it read 49,125 and created 319; the final unchanged turn read 49,444 and created 54. The comparable pre-fix post-tag request read zero and recreated 49,145 tokens. The fix therefore changed the cache mix from a full-prefix creation to a warm-prefix read, reducing cache creation by 48,826 tokens in that observed step. Total `contextTokens` did not fall—49,452 post-fix versus 49,155 in the non-identical pre-fix prompt—because `contextTokens` includes both cache reads and cache creation.

Removing `askSideQuestion` did not remove either invariant no-tools auxiliary request. Post-fix children still emitted two byte-identical auxiliary requests apiece, costing 632, 646, or 666 context tokens each depending on prompt length. Their identical request fingerprints make the earlier title-versus-tags attribution incorrect. The strongest current inference is that the removed side request perturbed the main lane's cache policy while the two remaining auxiliary requests belong to another naming/title path; private bodies are not logged, so their semantics remain unproven.

### Explicit SDK title-generation ablation

The next ablation removes only Nimbalyst's `Query.generateSessionTitle` call from the Claude provider. The host still persists its provisional first-prompt title before the provider starts, `hasOutOfBandNaming` remains enabled so the agent does not spend a tool call renaming the session, and the default phase fallback remains independent of any model request.

Two post-restart `claude-code:haiku` children completed read-only with no edited files. Each emitted exactly one auxiliary request rather than two:

| Task topic                         | auxiliary requests | auxiliary contextTokens | persisted title prefix                                                    |
| ---------------------------------- | -----------------: | ----------------------: | ------------------------------------------------------------------------- |
| Pinned collaboration tabs          |                  1 |                     630 | `Investigate whether pinned collaboration tabs survive app restarts...`   |
| Resumed-session tag cache behavior |                  1 |                     631 | `Review cache behavior when session tags change during a resumed Claude...` |

This removes one measured 630-631-token request from each first turn in the matched post-ablation runs. Metadata calls still succeeded, the main-lane system/tool/order/server fingerprints stayed stable, and both sessions retained task-relevant titles. The titles are the host's longer truncated first-prompt strings rather than concise model-generated summaries, so this is a measurable context saving with a title-quality tradeoff rather than an unqualified improvement. The one remaining auxiliary request is still semantically unidentified because the proxy intentionally stores no private body text.

### In-band naming follow-up (resolves the title-quality tradeoff without a cache cost)

The ablation above left `hasOutOfBandNaming` enabled, which suppresses the agent's own naming and leaves only the host's truncated first-prompt title. The follow-up recovers a real model-generated title without re-adding any request: claude-code now names the session in-band via the `update_session_meta` call the agent already makes for tags and phase (`hasOutOfBandNaming` flipped to false). No new auxiliary request, so the 630-631-token saving stands and the title-quality tradeoff is resolved.

There is a prompt-cache trap here that is easy to reintroduce, so it is recorded explicitly. The Nimbalyst system prompt is sent as `--append-system-prompt` and re-sent on every resumed turn, at the front of the cache prefix. The in-band vs out-of-band naming section is different text, and `hasBeenNamed` is mutable — a normal session is unnamed on turn 1 and named on turn 2. Gating the prompt on the live flag would flip the appended prompt between turns and force a full `system_changed` cache miss on turn 2 of every multi-turn session, re-billing the entire ~47k prefix uncached — far more than the 630 tokens saved.

The fix decides the naming mode once, on the session's first prompt build (which precedes the agent's in-band naming, so `hasBeenNamed` there means a caller-assigned title such as a spawn_session child), and freezes it for the session's life. A caller-titled session stays out-of-band so it does not clobber the parent's title; every other session names itself in-band. The invariant — appended prompt byte-identical across turns — is asserted by `ClaudeCodeProvider.sessionNaming.test.ts` (`turn2 === turn1`), which also confirms the addendum has no other per-turn non-determinism. Cache-stability was verified by that deterministic byte-equality assertion; the paid harness (`run.ts`) only measures first-turn structural cost and does not drive the provider across two turns, so it would not have exercised this path.

## Ranked opportunities

1. Keep the now-static `update_session_meta` schema and its fingerprint regression gate. Confidence: high. The live mutation test replaced a 49,145-token full-prefix creation with a 49,125-token read plus 319 tokens of incremental creation.
2. Keep tags/phase on the metadata tool path rather than the removed SDK side question. Confidence: high for cache stability. The exact sequential post-fix repeat preserved `ttl: 1h` and read the prior prefix after both `Read` and metadata continuations.
3. Keep the explicit SDK `generateSessionTitle` call disabled and name in-band via `update_session_meta` instead. Confidence: high for the 630-631-token measured saving and cache-neutrality; the in-band path restores a real model-generated title at no extra request, superseding the earlier truncated-title tradeoff. Must stay decided-once-per-session (see the in-band naming follow-up) so the appended prompt does not churn the cache.
4. Identify the remaining auxiliary request before attempting another removal. Confidence: high that one request remains, low on semantic ownership. It cost 630-631 tokens in the current pair, but private prompt contents are intentionally unavailable.
5. Freeze the MCP snapshot before the first request and across resume, or wait for the required eager surface before sending. Confidence: high. The observed pending-to-connected transition changed the API tool prefix and 7,344 actual first-turn tokens; in a long resumed session it can invalidate the entire cached conversation.
6. Keep ToolSearch enabled and add a regression guard. Confidence: high. It reduced Nimbalyst-specific overhead by 53,858 actual tokens in the matched Haiku comparison.
7. Reduce or conditionally expose extension MCP groups. Confidence: high for sizing, product tradeoff unresolved. Extension MCP names cost 3,529 actual tokens even while their schemas were deferred; platform MCP names cost 919.
8. Defer low-frequency eager tools. Confidence: medium until an actual-token ablation is run. `developer_git_commit_proposal`, `capture_editor_screenshot`, and `get_session_edited_files` total 2,602 schema bytes and about 651 estimated tokens, but no actual savings claim should be made from that estimate alone.
9. Continue compacting the addendum only where behavior remains protected by tests. Confidence: high for current size, medium for further cuts. The current compacted addendum still costs 1,258 actual tokens.
10. Treat plugin command/skill deduplication as low priority for first-turn context under 2.1.210. Confidence: high. The isolated cost was 69 actual tokens.

## Limitations

- Actual factorial attribution used Haiku 4.5, while the historical 13,072-token checkpoint used Fable. Byte measurements are current and exact; token counts are model-specific.
- Four-byte estimates are not tokenizer output and are not additive claims.
- `contextTokens` is `input + cache_creation + cache_read`; a cache hit can greatly reduce cache creation and billing without reducing this total. Cache creation/read are therefore reported separately for cache-stability fixes.
- The primary matrix's random session id intentionally avoided mutating a real session, but it omitted live tag counts. The separate real-session toggle measured that gap and restored session metadata afterward.
- The factorial matrix uses fresh first-turn calls. The live follow-up adds a resumed three-turn lineage, but it is still short and does not cover TTL expiry or compaction.
- One all-MCP run was pending and was excluded from stable component attribution. It is retained as evidence of a real connection-timing instability.
- The live proxy observes in-process SDK API traffic, including auxiliary requests, but intentionally stores no bodies. Auxiliary semantics are inferred from shape and interleaving rather than private prompt text.
- Request metadata changes between fresh CLI invocations but stayed fixed inside each live resumed session. Cross-session cache-key significance remains unknown.
- The title ablation used two descriptive English prompts. It does not yet cover terse prompts, slash commands, localized prompts, or cases where a concise semantic title matters more than the host's truncated first-prompt title.
